# Frameworks de Copywriting

## PAS (Problem - Agitate - Solution)

**Cuándo usar:** Cuando el producto resuelve un dolor claro y reconocible. Ideal para SaaS que eliminan tareas tediosas o resuelven frustraciones específicas.

### Estructura

```
PROBLEM: Identificar el problema específico que el usuario experimenta
AGITATE: Amplificar las consecuencias de no resolverlo (sin exagerar)
SOLUTION: Presentar el producto como la solución natural
```

### Ejemplo Aplicado

**Producto:** App de gestión de gastos para freelancers

```
PROBLEM:
Cada mes es lo mismo: facturas en el email, tickets en la cartera,
y una carpeta de "gastos pendientes" que da miedo abrir.

AGITATE:
Cuando llega la declaración trimestral, pasas 2 días reconstruyendo
qué gastaste y en qué. Siempre te falta algún recibo. Y la sensación
de que estás pagando más impuestos de los que deberías no se va.

SOLUTION:
[Nombre] fotografía tus tickets, los categoriza automáticamente y
genera el informe que tu gestor necesita. En 3 clics, no en 3 días.
```

### Errores comunes con PAS

- **Inventar problemas**: Si el usuario no reconoce el problema, el framework falla
- **Agitar demasiado**: Crear ansiedad innecesaria es manipulación, no persuasión
- **Solución desconectada**: La solución debe responder exactamente al problema planteado

## AIDA (Attention - Interest - Desire - Action)

**Cuándo usar:** Para productos nuevos o categorías desconocidas. Cuando necesitas educar al usuario sobre por qué debería importarle.

### Estructura

```
ATTENTION: Captar atención con algo inesperado o relevante
INTEREST: Generar curiosidad con información útil
DESIRE: Conectar beneficios con las necesidades del usuario
ACTION: Indicar el siguiente paso claro
```

### Ejemplo Aplicado

**Producto:** Herramienta de transcripción de reuniones con IA

```
ATTENTION:
El 73% de lo que se dice en reuniones se olvida en 24 horas.

INTEREST:
Tu equipo pasa 15 horas semanales en reuniones. Sin notas estructuradas,
las decisiones se pierden, los action items se olvidan, y la siguiente
reunión empieza repitiendo la anterior.

DESIRE:
[Nombre] transcribe tus reuniones, extrae los action items, y los asigna
automáticamente a cada persona. Nadie más dice "¿quedamos en qué?"

ACTION:
Conecta tu calendario y prueba con tu próxima reunión. Gratis.
```

### Errores comunes con AIDA

- **Attention falsa**: Datos inventados o exagerados destruyen credibilidad
- **Interest que no educa**: Si solo generas curiosidad sin dar valor, parece clickbait
- **Desire manipulativo**: Crear deseo artificial con escasez falsa o FOMO

## BAB (Before - After - Bridge)

**Cuándo usar:** Para mostrar transformación. Ideal cuando el beneficio es un cambio de estado claro (de manual a automático, de caos a orden, de lento a rápido).

### Estructura

```
BEFORE: La situación actual del usuario (sin el producto)
AFTER: La situación después de usar el producto
BRIDGE: Cómo el producto conecta ambos estados
```

### Ejemplo Aplicado

**Producto:** Dashboard de métricas para equipos de producto

```
BEFORE:
Cada lunes empiezas igual: abriendo Mixpanel, Google Analytics,
el CRM y tres hojas de cálculo. Una hora después, tienes los números.
Desactualizados para el miércoles.

AFTER:
Todos los datos de tu producto en un solo lugar. Actualizados en tiempo
real. Tu equipo toma decisiones con datos de hoy, no de la semana pasada.

BRIDGE:
[Nombre] se conecta a tus herramientas en 5 minutos. Sin código, sin
migraciones, sin consultores. Solo las métricas que importan, cuando las necesitas.
```

### Errores comunes con BAB

- **Before irreal**: Exagerar el dolor actual pierde credibilidad
- **After fantasioso**: Prometer resultados que el producto no puede entregar
- **Bridge vago**: "Nuestra solución innovadora" no explica nada

## 4Ps (Promise - Picture - Proof - Push)

**Cuándo usar:** Para landing pages de conversión directa. Cuando tienes prueba social sólida y quieres ir directo al grano.

### Estructura

```
PROMISE: Beneficio principal en una línea
PICTURE: Visualización de cómo será usar el producto
PROOF: Evidencia de que funciona
PUSH: CTA con urgencia legítima (si existe)
```

### Ejemplo Aplicado

**Producto:** API de pagos para marketplaces

```
PROMISE:
Cobra a tus vendedores y paga a tus compradores. Sin ser un banco.

PICTURE:
Tu marketplace funcionando como MercadoLibre o Airbnb: split payments
automáticos, retenciones configurables, y payouts programados. Todo
desde tu propio dashboard.

PROOF:
+200 marketplaces en LATAM procesan $50M mensuales con [Nombre].
Incluyendo [Logo 1], [Logo 2], [Logo 3].

PUSH:
Integra en tu sandbox hoy. Documentación en español, soporte en tu zona horaria.
```

## Cuándo NO Usar Frameworks

Los frameworks son guías, no plantillas rígidas. Evitar usarlos cuando:

1. **El producto es simple**: Si la propuesta de valor es obvia, un headline directo funciona mejor que una estructura elaborada
2. **El usuario ya sabe qué busca**: Para búsquedas de alta intención, ir directo a features/pricing
3. **Hay poco espacio**: En ads o microcopy, la brevedad gana

## Combinación de Frameworks

Es válido combinar elementos. Por ejemplo:

- Hero con BAB + Sección de features con AIDA
- PAS para el problema + 4Ps para la conversión

Lo importante es que el copy fluya naturalmente, no que siga una estructura por obligación.
